## Overview

A typical nene.js project follows a structured layout that separates concerns while maintaining the unified development experience:

```text
my-nene-app/
├── src/
│   ├── app/              # Next.js App Router pages
│   ├── controllers/      # nene.js controllers
│   ├── services/         # Business logic services
│   ├── hooks/            # Auto-generated hooks
│   └── components/       # React components
├── nene.config.ts        # nene.js configuration
├── next.config.js        # Next.js configuration
└── package.json
```

## Key Directories

### `src/controllers/`

Contains your nene.js controllers. These define your API endpoints and generate corresponding frontend hooks.

```typescript
// src/controllers/post.controller.ts
@Controller("/api/posts")
export class PostController {
  @Get("/")
  async getPosts() {
    /* ... */
  }

  @Post("/")
  async createPost(@Body() data: CreatePostDto) {
    /* ... */
  }
}
```

<Callout type="info">
  Controllers are automatically discovered and registered when placed in this
  directory.
</Callout>

### `src/hooks/`

Auto-generated directory containing type-safe React hooks for each controller method.

<Callout type="warning">
  Don't edit files in this directory manually - they will be overwritten when
  hooks are regenerated.
</Callout>

### `src/services/`

Business logic that can be shared between controllers. Uses dependency injection:

```typescript
// src/services/user.service.ts
import { Injectable } from "@nene/core";

@Injectable()
export class UserService {
  async findById(id: string) {
    // Business logic here
  }
}
```

### `src/app/`

Standard Next.js App Router directory for your React pages and layouts.

## Configuration Files

| File             | Description                     |
| ---------------- | ------------------------------- |
| `nene.config.ts` | nene.js framework configuration |
| `next.config.js` | Next.js configuration           |
| `tsconfig.json`  | TypeScript configuration        |

## Best Practices

1. **Group by feature** - For larger apps, consider grouping controllers and services by feature
2. **Keep controllers thin** - Move business logic to services
3. **Use DTOs** - Define clear data transfer objects for validation
